    BML Script API Shape and Coverage Redesign Plan¶

    This document defines the next clean-break direction for the BML+ script-facing API. The goal is broad practical coverage without turning the script API into a large object hierarchy or a raw native ABI dump.

    The current surface is functional, but it has two structural problems:

    • BML:: global functions and BML::ModContext mirror most of the same runtime operations. The current stub has roughly 130 ModContext methods, roughly 130 global functions, and about 100 duplicate names.
    • Important author workflows are still shallow or missing: physics helpers, object loading, DataArray access, richer config, richer timer semantics, and safer content registration definitions.
    • The script mod callback shape needs a semantic contract. It should not mirror the native IMod ABI one method at a time, but it also should not collapse everything into one dynamic OnEvent catch-all.

    Design Principles¶

    • Keep the API thin. Do not introduce a deep service tree such as ctx.World.Objects.Materials.
    • Make ownership explicit. Operations owned by the current script mod must start from ModContext.
    • Keep true utilities global. Global BML:: functions should not depend on the current script mod owner or active callback.
    • Keep raw CK access available, but honest. Borrowed CK handles use Borrow* names and are documented as callback/runtime-scope escape hatches.
    • Prefer small definition objects over long positional parameter lists.
    • Prefer stateless helper namespaces over per-CK-object wrapper classes.
    • Do not expose CKAngelScript internals, asIScript*, raw callbacks, void*, native userData, raw Bui lifetime classes, or behavior hook callbacks to ordinary scripts.

    Target Shape¶

    The script API should be organized into a small number of shallow entry points:

    namespace BML {
  // Version, errors, pure path helpers, and optional borrowed current context.
}

class BML::ModContext {
  // Current script mod owner-aware API.
}

namespace BML::CK {
  // Stateless helpers for borrowed CK handles.
}

namespace BML::Physics {
  // Safe ExecuteBB-style physics actions.
}

namespace BML::Text {
  // Safe text and font helpers.
}

namespace BML::UI {
  // Stable Bui-style immediate menu facade.
}

namespace ImGui {
  // Advanced generated frame-scope ImGui API.
}

    This keeps the API flat enough for authors while preventing the current BML::Foo() / ctx.Foo() duplication.

    Global BML:: Surface¶

    Global functions should be limited to context-free utilities:

    namespace BML {
  string GetVersion();
  int GetVersionMajor();
  int GetVersionMinor();
  int GetVersionPatch();
  string GetErrorString(int code);

  bool BorrowCurrentContext(ModContext &out context);

  namespace Path {
    bool Exists(const string &in path);
    bool IsFile(const string &in path);
    bool IsDirectory(const string &in path);
    bool IsValid(const string &in path);
    bool IsAbsolute(const string &in path);
    bool IsRelative(const string &in path);
    string Combine(const string &in left, const string &in right);
    string Normalize(const string &in path);
    string FileName(const string &in path);
    string Extension(const string &in path);
    string RemoveExtension(const string &in path);
  }
}

    BorrowCurrentContext() exists only for low-level boundaries where BML calls script code without an explicit context parameter. It returns a borrowed call-scope context and must not appear in ordinary examples. It returns false outside a BML-owned script callback/export/timer/command/DataShare invocation.

    The following global functions should be removed in the clean break:

    • owner-aware runtime operations such as AddTimer, RegisterCommand, RequestDataShare, SendIngameMessage, and ExecuteCommand;
    • named CK object lookup functions such as Get3dEntityByName;
    • object helper functions such as GetObjectName;
    • content registration functions such as RegisterBallType;
    • HUD/menu/game-state helpers that naturally belong to the active mod context.

    Script Mod Event Contract¶

    Script mods should be semantically complete BML mods, not raw IMod ABI mirrors. The callback surface should be shaped by event semantics:

    • Keep special lifecycle callbacks explicit.
    • Aggregate only homogeneous event families.
    • Keep structured payload events strongly typed.
    • Do not introduce a universal OnEvent object with Kind and As*() projections.

    A single universal OnEvent looks tidy but is the wrong AngelScript shape. It creates a hand-written dynamic type system: scripts must switch on a kind, call nullable projection helpers, and remember which fields are valid for which payload. That is worse for editor stubs, static validation, documentation, and callback lifetime rules.

    Callback Groups¶

    Core lifecycle callbacks stay explicit because they have special frequency, failure, and frame-scope semantics:

    void OnLoad(const BML::ModContext &in ctx);
void OnUnload(const BML::ModContext &in ctx);
void OnProcess(const BML::ModContext &in ctx);
void OnRender(const BML::ModContext &in ctx,
              const BML::RenderEvent &in event);

    Homogeneous event families stay aggregated:

    void OnGameEvent(const BML::ModContext &in ctx,
                 BML::GameEvent event);
void OnCommandEvent(const BML::ModContext &in ctx,
                    const BML::CommandEvent &in event);

    OnGameEvent is for no-payload game lifecycle signals such as start menu, level load, checkpoint, death, and life changes. OnCommandEvent is for the global command bus; pre/post phases and script-owned execute/complete phases share the same event type because they are one domain model.

    Structured payload events stay explicit:

    void OnModifyConfig(const BML::ModContext &in ctx,
                    const BML::ConfigEvent &in event);
void OnLoadObject(const BML::ModContext &in ctx,
                  const BML::LoadObjectEvent &in event);
void OnLoadScript(const BML::ModContext &in ctx,
                  const BML::LoadScriptEvent &in event);
void OnPhysicalize(const BML::ModContext &in ctx,
                   const BML::PhysicalizeEvent &in event);
void OnUnphysicalize(const BML::ModContext &in ctx,
                     const BML::ObjectEvent &in event);

    Decision rule:

    Event shape Script callback shape
    High-frequency or special execution scope Dedicated callback
    No-payload homogeneous signal family Aggregated enum event
    Same domain bus with phases Aggregated typed event
    Structured payload with distinct resources/lifetime Dedicated typed event
    Requires cancellation or mutation Separate intercept design, not retrofitted onto readonly events

    Command Events¶

    CommandEvent remains the single command-domain payload:

    enum CommandEventPhase {
  COMMAND_EVENT_PRE,
  COMMAND_EVENT_POST,
  COMMAND_EVENT_EXECUTE,
  COMMAND_EVENT_COMPLETE
}

class CommandEvent {
  CommandEventPhase get_Phase() const;
  bool get_IsPre() const;
  bool get_IsPost() const;
  bool get_IsExecute() const;
  bool get_IsComplete() const;
  string get_CommandName() const;
  int get_ArgCount() const;
  string GetArg(int index) const;
  string get_ArgsText() const;
  bool get_IsCheat() const;
}

    The global OnCommandEvent sees pre/post execution phases. Script-owned Command.Execute and Command.Complete reuse the same event type with execute/complete phases. If command cancellation, argument rewriting, or result replacement is needed later, add a separate intercept API rather than mutating this readonly event.

                          CKMesh@ mesh = null);
  bool Unphysicalize(CK3dEntity@ target);
  bool SetForce(CK3dEntity@ target,
                const VxVector &in position,
                CK3dEntity@ positionReference,
                const VxVector &in direction,
                CK3dEntity@ directionReference,
                float force);
  bool ClearForce(CK3dEntity@ target);
  bool Impulse(CK3dEntity@ target,
               const VxVector &in position,
               CK3dEntity@ positionReference,
               const VxVector &in direction,
               CK3dEntity@ directionReference,
               float impulse);
  bool WakeUp(CK3dEntity@ target);
}

    Physics helpers are runtime-level operations. They return false without crashing when the BML runtime is unavailable, the game is outside a loaded level, the target handle is null/stale, or native ExecuteBB rejects the operation. Failures should record a script diagnostic with enough context to identify the operation and target name/id when available.

    Object loading needs a result wrapper because the native return type includes an XObjectArray *.

    class ObjectLoadOptions {
  string File;
  bool Rename;
  string MasterName;
  int FilterClass;
  bool AddToScene;
  bool ReuseMeshes;
  bool ReuseMaterials;
  bool Dynamic;
}

class ObjectLoadResult {
  bool get_Success() const;
  CKObject@ BorrowMainObject() const;
  int get_Count() const;
  CKObject@ BorrowObject(int index) const;
}

namespace BML::CK {
  ObjectLoadResult@ LoadObject(const ObjectLoadOptions &in options);
}

    ObjectLoadResult is an owning script object for the native temporary load result. It must not expose or retain XObjectArray * directly. The implementation should copy the loaded object IDs into the result and release the native temporary array immediately. BorrowMainObject() and BorrowObject() resolve those IDs back to borrowed CK handles on demand and return null after the object disappears.

    Do not expose CreateHookBlock or native behavior callbacks.

    Timer Object Contract¶

    The current timer object direction is correct, but the contract should cover the native timer model more completely.

    The native registration surface should use an AngelScript interface. A pure script convenience base class may exist in the author stub/templates, but BML must register and validate against the interface so ownership and dispatch rules stay explicit.

    interface Timer {
  string get_Name() const;
  int get_Type() const;
  int get_TimeBase() const;
  int get_DelayTicks() const;
  float get_DelaySeconds() const;
  int get_RepeatCount() const;
  int get_Priority() const;
  bool get_StartPaused() const;
  string get_Group() const;

  bool Tick(const BML::ModContext &in ctx,
            const BML::TimerEvent &in event);
  void Progress(const BML::ModContext &in ctx,
                const BML::TimerEvent &in event,
                float progress);
}

    TimerRef should keep the existing generation-checked semantics and add group or timing metadata only if it can be queried without allocations on the hot path. AddTimer(Timer@+ timer) retains the script object exactly once on successful registration and releases it on cancel, completion, or mod unload. Failed registration does not consume the incoming @+ handle.

    UI and ImGui¶

    BML::UI remains the preferred ordinary author API. It should cover Bui-style render-time controls, but not Bui lifetime classes.

    Add or verify:

    • OkButton;
    • KeyButton;
    • RadioButtonText using newline-separated choices;
    • InputTextButton(string &inout);
    • key chord formatting and capture helpers;

    Migration Plan¶

    Phase 1: Stop the Duplication¶

    • Add the new declarations to docs/bml-script-mod-api.as behind the intended clean-break surface.
    • Remove owner-aware global functions from the public stub and registration table.
    • Keep only version/error/path/borrow-current-context and global typed DataShare get/set functions.
    • Rename raw lookup functions to Borrow*.
    • Move object helper functions from ModContext and global BML:: into BML::CK.

    Phase 2: Fill Practical Coverage¶

    • Replace typedef int GameEvent with a real GameEvent enum and add GetGameEventName(GameEvent).
    • Add OnModifyConfig and ConfigEvent.
    • Keep OnGameEvent and OnCommandEvent as the intentional aggregated event shapes. Do not introduce a universal OnEvent catch-all.
    • Add BML::CK DataArray helpers.
    • Add BML::Physics wrappers over safe ExecuteBB actions.
    • Add ObjectLoadOptions / ObjectLoadResult.
    • Add Text2DDefinition / BML::Text.
    • Add full config category/property helpers.

    Phase 3: Enrich Event Views¶

    • Extend LoadObjectEvent with object count, object id lookup, borrowed object lookup, and borrowed master object access.
    • Extend LoadScriptEvent with BorrowScript().
    • Extend PhysicalizeEvent with borrowed target, convex mesh access, ball center/radius access, and concave mesh access.
    • Extend ObjectEvent with borrowed target access.
    • Keep all event views callback-scope and document durable identity via CK ids or CKAS ObjectRef@-derived handles.

    Phase 4: Make Object APIs Explicit¶

    • Replace minimal Command, Timer, and DataShareRequest interfaces with complete interfaces. Optional pure-script base classes may be provided in examples, but the native registration surface stays interface-based.
    • Keep @+ ownership declarations for APIs that retain script objects.
    • Add ownership smoke tests for temporary object registration, failed registration, self-cancel/unregister, and unload invalidation.

    Phase 5: Documentation and Tests¶

    • Update the author guide so ordinary examples use only ctx, BML::CK, BML::Physics, BML::UI, and ImGui in their intended roles.
    • Update the coverage document from native-method parity to task/domain coverage.
    • Add smoke coverage for OnGameEvent, OnCommandEvent, OnModifyConfig, enriched load/physics event views, physics, object loading, DataArray, config categories, Timer variants, Command completion, DataShare request, UI, and ImGui.
    • Keep the negative check forbidding CKAS/asIScript/raw callback/userData leaks.

    Validation Requirements¶

    • BML_ENABLE_ANGELSCRIPT=ON MSVC x86 build passes.
    • BML_ENABLE_ANGELSCRIPT=OFF MSVC x86 build passes.
    • No old owner-aware global function declarations remain in the script stub.
    • No ordinary script declaration exposes CKAngelScript handles, asIScript*, void*, native callbacks, or userData.
    • No Bui Window, Page, or Menu classes are exposed.
    • Trigger paths for Timer, Command, DataShare, exports, and callbacks do not do post-load method lookup.
    • No universal OnEvent catch-all is exposed for BML script mod callbacks.
    • GameEvent is a real enum and OnGameEvent remains no-payload only.
    • Structured callback payloads expose typed event views, not raw native ABI pointers.
    • Ballance Player smoke covers single-file, directory, zip script packages, game events, command events, config modification, enriched object/physics events, physics, object load, DataArray, Timer, Command, DataShare, exports, UI, and ImGui.

    Deferred¶

    • Hot reload.
    • Script package signing and permission prompts.
    • Full CK object ownership wrappers.
    • Raw behavior graph editing.
    • Native BB hook callbacks.
    • Raw CKAngelScript escape hatch for ordinary scripts.
